# บทที่ 3: เข้าใจลูกค้าของคุณ

> *จำนวนเงินทั้งหมดใน playbook นี้เป็นสกุล USD (ดอลลาร์สหรัฐ) ยกเว้นระบุไว้เป็นอย่างอื่น*

---

Project ที่ล้มเหลวไม่ได้ล้มเพราะเทคโนโลยีไม่ดี แต่ล้มเพราะคนสร้างไม่เคยเข้าใจคนอีกฝั่งอย่างแท้จริง

คุณเจอปัญหาที่คุ้มค่าแก่การแก้แล้ว พิสูจน์แล้วว่าคนจริงมีปัญหานี้และยินดีจ่ายเพื่อแก้ ทีนี้มาถึงขั้นตอนที่คนสร้างส่วนใหญ่ข้าม — ขั้นตอนที่กำหนดว่าวิธีแก้ปัญหาของคุณจะถูกนำไปใช้จริง หรือจะนอนอยู่ในโฟลเดอร์ demo เก็บฝุ่น

คุณต้องเข้าใจคนที่จะซื้อ ใช้ อนุมัติ และอาจขัดขวางวิธีแก้ปัญหาของคุณอย่างลึกซึ้ง ไม่ใช่ "ผู้ใช้" ในเชิงนามธรรม แต่เป็นคนจริงที่มีความกลัว แรงจูงใจ และกระบวนการตัดสินใจเฉพาะตัว

บทนี้เกี่ยวกับการฟังก่อนสร้าง เข้าใจสิ่งที่ธุรกิจต้องการจริงๆ — ไม่ใช่สิ่งที่คุณคิดว่าเขาต้องการ เชื่อมโยงความเจ็บปวดกับคุณค่า เขียน value proposition ในภาษาของเขา และสร้างความเข้าใจตรงกันที่จะเปลี่ยนไอเดียที่พิสูจน์แล้วเป็น project ที่ได้รับเงินสนับสนุน

---

## 3.1 ทำไมความเข้าใจต้องมาก่อนการสร้าง

สาเหตุอันดับหนึ่งที่ทำให้ project AI automation ล้มเหลวไม่ใช่เรื่องเทคนิค แต่เป็นเรื่องความไม่ตรงกัน

Developer สร้างสิ่งที่ตัวเองคิดว่าลูกค้าต้องการ ลูกค้าต้องการอะไรที่ต่างไปเล็กน้อย ช่องว่างไม่ได้ใหญ่โต — อาจจะคลาดกันแค่ 20% แต่ 20% นั้นคือความแตกต่างระหว่าง "สิ่งนี้เปลี่ยนแปลงธุรกิจผม" กับ "น่าสนใจดี แต่เราเลิกใช้หลังจากผ่านไปเดือนเดียว"

ผมเห็นรูปแบบนี้ซ้ำแล้วซ้ำเล่า:

- Developer สร้าง bot คัดกรอง lead ที่ถามคำถามคัดกรองผิด — เพราะตั้งสมมติฐานเองว่าอะไรสำคัญสำหรับทีมขาย แทนที่จะถาม
- Automation ส่งเตือนนัดหมายตอน 8 โมงเช้า เพราะ developer คิดว่าสมเหตุสมผล แต่คนไข้ของคลินิกชอบรับเตือนช่วงเย็นมากกว่า
- ระบบประมวลผลเอกสารส่งข้อมูลในรูปแบบที่ไม่ตรงกับโครงสร้าง spreadsheet ที่ลูกค้าใช้อยู่ กลายเป็นสร้างงาน manual เพิ่มแทนที่จะลดลง

ทุกข้อผิดพลาดเหล่านี้ป้องกันได้ด้วยการใช้เวลาสองชั่วโมงคุยก่อนเขียน code สักบรรทัด

### สองชั่วโมงที่ช่วยประหยัดสองเดือน

- **ไม่มีความเข้าใจลึกซึ้ง:** สร้างใน 2 สัปดาห์ ลูกค้าใช้เดือนหนึ่ง ขอแก้ไข สร้างใหม่ ขอแก้อีก สุดท้ายลูกค้ายอมรับวิธีแก้ปัญหาแบบพอใช้ได้ หรือยกเลิก เวลาทั้งหมด: 2 เดือน ความพึงพอใจ: ปานกลางในกรณีที่ดีที่สุด

- **มีความเข้าใจลึกซึ้ง:** ใช้เวลา 2 ชั่วโมงคุยก่อนสร้าง สร้างใน 2 สัปดาห์ — แต่สร้างถูกอย่าง ปรับเล็กน้อย ไม่ใช่สร้างใหม่ เวลาทั้งหมด: 3 สัปดาห์ ความพึงพอใจ: สูง

สองชั่วโมงในการฟัง หรือสองเดือนในการเดา นั่นคือทางเลือกจริงๆ

## 3.2 วิธีฟังสิ่งที่ธุรกิจต้องการจริงๆ

คนสร้างส่วนใหญ่ฟังเพื่อจับ requirement ผู้ดำเนินการที่เก่งฟังเพื่อจับความเจ็บปวด ความกลัว และแรงจูงใจ สองอย่างนี้ต่างกันมาก

### Requirement กับ ความเจ็บปวด

เมื่อผู้จัดการสำนักงานทันตกรรมบอกว่า "เราต้องการระบบส่งเตือนนัดหมาย" นั่นคือ requirement เป็นสิ่งที่เขาคิดว่าวิธีแก้ปัญหาควรมีหน้าตาอย่างไร

แต่ถ้าถามลึกขึ้น — "ช่วยเล่าให้ฟังหน่อยว่าเกิดอะไรขึ้นเมื่อคนไข้ไม่มาตามนัด" — คุณอาจได้ยินว่า: "เมื่อคนไข้ไม่มา หมอฟันมีช่องว่าง 45 นาทีที่ไม่มีรายได้ ถ้าเกิดขึ้นวันละสองครั้ง เราสูญเสียรายรับ $400-600 แถมพนักงาน front desk ก็เครียดเพราะรู้สึกว่าเป็นความรับผิดชอบของตัวเอง"

ตอนนี้คุณเข้าใจความเจ็บปวดแล้ว Requirement คือ "ส่งเตือน" ความเจ็บปวดคือรายได้ที่หายไป พนักงานเครียด และหมอที่หงุดหงิด วิธีแก้ปัญหาที่ตอบโจทย์ความเจ็บปวดอาจมีหน้าตาต่างออกไป — อาจรวมระบบ waitlist ที่เติมช่องว่างจากการยกเลิกนัด ไม่ใช่แค่ส่งเตือน

**ความเจ็บปวดเผยให้เห็นคุณค่า Requirement เผยให้เห็น feature** คุณตั้งราคาจากคุณค่า ไม่ใช่จาก feature

### ความกลัวและแรงจูงใจ

ทุกคนที่เกี่ยวข้องกับการตัดสินใจซื้อมีความกลัวและแรงจูงใจที่ขับเคลื่อนพฤติกรรม การเข้าใจสิ่งเหล่านี้คือความแตกต่างระหว่างการปิดการขายในการประชุมเดียว กับการเสียเขาไปทั้งหมด

**เจ้าของธุรกิจกลัว:**
- เสียเงินไปกับเทคโนโลยีที่ใช้ไม่ได้
- ต้องพึ่งพา vendor ที่อาจหายไป
- พนักงานปฏิเสธระบบใหม่
- ดูเสียหน้าถ้า project ล้มเหลว

**แรงจูงใจของเจ้าของธุรกิจ:**
- ลดต้นทุน (ซึ่งเชื่อมตรงกับรายได้ส่วนตัวในธุรกิจขนาดเล็ก)
- มีเวลาว่างมากขึ้น (เจ้าของธุรกิจขนาดเล็กหลายคนทำงาน operation เองเพราะจ้างคนเพิ่มไม่ไหว)
- ไม่ตกยุค (เห็นคู่แข่งเริ่มใช้เทคโนโลยี)

**ผู้จัดการฝ่ายปฏิบัติการกลัว:**
- ต้องเรียนรู้ระบบใหม่ (ตอนนี้ก็ล้นมือแล้ว)
- ถูกแทนที่ด้วย automation (ความกลัวที่ไม่มีใครพูดถึง)
- ถูกโทษถ้า automation พัง
- สูญเสียการควบคุมกระบวนการ "ของตัวเอง"

**แรงจูงใจของผู้จัดการฝ่ายปฏิบัติการ:**
- งานน่าเบื่อน้อยลง
- ได้รับการยอมรับจากการปรับปรุงกระบวนการ
- ไม่ต้องถูกโทรมาวันเสาร์อาทิตย์เพราะมีอะไรตกหล่น

ในวัฒนธรรมธุรกิจเอเชียตะวันออกเฉียงใต้ มีอีกมิติที่สำคัญมาก: "หน้า" (face) ในวัฒนธรรมไทย หน้าเป็นแนวคิดทางสังคมที่สำคัญมาก การทำให้ใครเสียหน้า — แม้โดยไม่ตั้งใจ — สามารถทำลายความสัมพันธ์ทางธุรกิจได้ทันที

อย่าวางกรอบ automation ว่าเป็นการเปิดเผยปัญหาในกระบวนการปัจจุบัน หรือสื่อว่าทีมทำอะไรผิดมาตลอด

**วางกรอบผิด:** "กระบวนการปัจจุบันของคุณไม่มีประสิทธิภาพ คุณเสียเวลา 20 ชั่วโมงต่อสัปดาห์กับงาน manual"

**วางกรอบถูก:** "ทีมของคุณรับมือกับ volume ได้เยอะมาก AI ช่วยสนับสนุนพวกเขาได้โดยดูแลส่วนที่ซ้ำซาก ให้ทีมได้โฟกัสกับงานที่ต้องใช้วิจารณญาณของคน"

ข้อความเดียวกัน กรอบต่างกัน แบบแรกทำให้เสียหน้า แบบที่สองให้หน้า การวางกรอบแบบนี้สำคัญทุกที่ แต่ใน SEA มันคือความแตกต่างระหว่างการ rollout ที่ราบรื่นกับการบ่อนทำลายแบบเงียบๆ เมื่อทีมรู้สึกว่าได้รับการสนับสนุนแทนที่จะถูกคุกคาม เขาจะร่วมมือกับการ rollout แทนที่จะขัดขวาง

**VP ในองค์กรกลัว (Track 4 — project ภายใน):**
- Budget บานปลาย
- เรื่องความปลอดภัยหรือ compliance
- เสียหน้าถ้า project ล้มเหลวอย่างเปิดเผย
- สร้างบรรทัดฐาน ("ถ้าแผนกคุณได้ budget AI ทุกคนก็จะอยากได้")

**แรงจูงใจของ VP ในองค์กร:**
- ตี KPI (ลดต้นทุน เพิ่มประสิทธิภาพ ปรับจำนวนพนักงานให้เหมาะสม)
- ดูทันสมัยต่อหน้าผู้บริหาร
- สร้าง track record ของโครงการที่ประสบความสำเร็จ

เมื่อเข้าใจทั้งความกลัวและแรงจูงใจ คุณสามารถนำเสนอวิธีแก้ปัญหาที่ตอบโจทย์ทั้งสองด้าน "Automation นี้ประหยัดเวลา 20 ชั่วโมงต่อสัปดาห์" ตอบแรงจูงใจ "เราจะทำ pilot 2 สัปดาห์โดยไม่มีข้อผูกมัด เพื่อให้คุณเห็นผลลัพธ์ก่อนตัดสินใจ" ตอบความกลัว

ต้องพูดกันตรงๆ: คนไม่ได้ซื้อเทคโนโลยี คนซื้อความปลอดภัยจากความกลัว และความก้าวหน้าไปสู่สิ่งที่ต้องการ ยิ่งคุณซึมซับเรื่องนี้ได้เร็วเท่าไหร่ ก็ยิ่งดีเท่านั้น

### ความกลัวเรื่อง AI ทำผิดพลาด

มีความกลัวที่ครอบคลุมทุกบทบาทในการตัดสินใจซื้อ และเป็นเรื่องที่คนไม่ค่อยพูดออกมาตรงๆ: จะเกิดอะไรขึ้นเมื่อ AI ทำผิด?

เจ้าของธุรกิจนึกภาพใบแจ้งหนี้ผิดถูกส่งไปหาลูกค้าผิดคน ผู้จัดการฝ่ายปฏิบัติการจินตนาการว่าข้อความเตือนคนไข้ถูกส่งไปหาคนที่ยกเลิกนัดแล้ว VP เห็นภาพอีเมลที่ AI สร้างขึ้นทำให้บริษัทเสียหน้าต่อสาธารณะ

ทุกคนมีฉากจำลองที่แย่ที่สุดของตัวเอง — และมักจะไม่พูดออกมาตรงๆ

ความกลัวนี้สมเหตุสมผล AI จะทำผิดพลาด งานของคุณไม่ใช่การแกล้งทำเป็นว่าไม่มีทาง — แต่คือการแสดงให้เห็นว่าระบบของคุณจับข้อผิดพลาดก่อนที่จะถึงมือใคร

**วิธีจัดการกับความกลัวเรื่อง AI ทำผิดพลาด:**
- อธิบายเรื่อง confidence threshold: "ระบบจะทำงานอัตโนมัติเฉพาะเมื่อมั่นใจสูง สิ่งที่ไม่แน่ใจจะส่งเข้าคิวให้คนตรวจสอบ"
- แสดง audit trail: "ทุกการกระทำถูกบันทึกไว้ คุณดูได้ว่าระบบทำอะไรและทำไม"
- อธิบาย safety net: "สองสัปดาห์แรก ระบบทำงานใน shadow mode — ประมวลผลทุกอย่างแต่ไม่ลงมือทำ ทีมของคุณตรวจสอบทุก output ก่อนเปิดใช้งานจริง"
- เสนอปุ่มหยุด: "คุณหยุด automation ได้ทุกเมื่อด้วยข้อความเดียว"

เมื่อคุณจัดการกับความกลัวนี้ตรงๆ คุณแยกตัวเองออกจาก AI vendor ทุกรายที่โบกมือแล้วบอกว่า "AI ของเราแม่นยำมาก" คำกล่าวอ้างเรื่องความแม่นยำเป็นนามธรรม การอธิบายมาตรการป้องกันเป็นรูปธรรม

## 3.3 Framework การสัมภาษณ์สำหรับลูกค้าภายนอก

ถ้าคุณขายให้ธุรกิจภายนอกองค์กร (Track 1, 2 และ 3) นี่คือวิธีสัมภาษณ์ลูกค้าเชิงลึก การสัมภาษณ์เชิงลึกเลยจุดที่คุณ validate ใน discovery call บทที่ 2 ไปแล้ว คุณยืนยันแล้วว่าปัญหามีจริง ตอนนี้กำลัง map ภาพทั้งหมด: ทุก stakeholder ทุกความกลัว ทุกข้อจำกัด

### People Map

ก่อนสัมภาษณ์ใคร ให้ map ว่าใครเกี่ยวข้องกับการตัดสินใจซื้อ:

| บทบาท | ตัวอย่าง | สิ่งที่เขาสนใจ |
|------|---------|---------------------|
| **ผู้ซื้อเชิงเศรษฐกิจ** | เจ้าของธุรกิจ, หุ้นส่วน | ROI, การลดต้นทุน, ความเสี่ยง |
| **ผู้ซื้อเชิงเทคนิค** | คน IT, ผู้จัดการสำนักงาน | การ integrate, ความน่าเชื่อถือ, การ support |
| **ผู้ใช้งานจริง** | พนักงานที่ทำงาน manual อยู่ | ใช้งานง่าย, ความมั่นคงในตำแหน่งงาน, ปริมาณงาน |
| **ผู้มีอิทธิพล** | ที่ปรึกษาที่เชื่อถือ, นักบัญชี, เพื่อนร่วมอุตสาหกรรม | คุณน่าเชื่อถือหรือไม่ |
| **ผู้ขัดขวาง** | ใครก็ตามที่ได้ประโยชน์จากสถานะปัจจุบัน | ไม่อยากสูญเสียบทบาทหรืออำนาจ |

ในธุรกิจขนาดเล็ก (ต่ำกว่า 50 พนักงาน) ผู้ซื้อเชิงเศรษฐกิจและผู้ซื้อเชิงเทคนิคมักเป็นคนเดียวกัน ในองค์กรที่ใหญ่กว่า เป็นคนละคน และคุณต้องทำให้ทั้งสองฝ่ายพอใจ

**ข้อผิดพลาดสำคัญที่ต้องหลีกเลี่ยง:** ขายให้แค่คนที่ตอบข้อความของคุณ คนนั้นอาจกระตือรือร้นแต่ไม่ใช่คนตัดสินใจ ถามเสมอ: "ใครอีกบ้างที่ต้องมีส่วนร่วมในการตัดสินใจแบบนี้?"

### การสัมภาษณ์เชิงลึก (30-45 นาที)

**ช่วงที่ 1: สถานะปัจจุบัน (10 นาที)**

คุณครอบคลุมพื้นฐานของ workflow เขาใน discovery call แล้ว ตอนนี้ขุดลึกขึ้น — map ขั้นตอนที่แน่ชัด การส่งงานต่อ และเครื่องมือที่ใช้ ฟังหาวิธีแก้ปัญหาเฉพาะหน้าและการแก้ไข manual ที่เขาไม่ได้พูดถึงครั้งแรก

จดบันทึกด้วยคำศัพท์ที่เขาใช้ ถ้าเขาบอกว่า "spreadsheet สีน้ำเงิน" ก็จดว่า "spreadsheet สีน้ำเงิน" ถ้าเขาเรียกกระบวนการว่า "ตรวจสอบรายวัน" ก็จดว่า "ตรวจสอบรายวัน" คุณจะต้องใช้คำศัพท์ของเขาเวลานำเสนอวิธีแก้ปัญหา

**ช่วงที่ 2: ความเจ็บปวดและผลกระทบ (10 นาที)**

Discovery call ดึงความเจ็บปวดหลักออกมาแล้ว ตอนนี้ให้วัดผลให้ชัดเจน:

- ผลกระทบทางการเงินของความผิดพลาด (รายได้ที่สูญเสีย ค่าความผิดพลาด ค่าล่วงเวลา)
- ผลกระทบทางอารมณ์ต่อทีม
- ผลกระทบต่อลูกค้า
- ความถี่ที่เกิดปัญหา

จากนั้นถาม: "ถ้าใช้ไม้กายสิทธิ์เปลี่ยนอะไรได้อย่างเดียวเกี่ยวกับกระบวนการนี้ จะเลือกอะไร?" คำตอบเผยลำดับความสำคัญจริงของเขา — อาจไม่ใช่สิ่งที่คุณคาดคิด

**ช่วงที่ 3: ภูมิทัศน์การตัดสินใจ (10 นาที)**

"ถ้าเราจะลองหาวิธีแก้ปัญหาเรื่องนี้ ช่วยเล่าให้ฟังหน่อยว่ากระบวนการตัดสินใจในองค์กรของคุณเป็นอย่างไร"

ฟังเพื่อจับ:
- ใครอนุมัติ budget
- ช่วง budget ที่เป็นไปได้ ("เคยลงทุนกับเครื่องมือแบบนี้มาก่อนไหม?")
- อะไรจะทำให้ตอบตกลง — และอะไรจะทำให้ตอบปฏิเสธ
- ความคาดหวังเรื่อง timeline

"อะไรจะทำให้คุณมั่นใจพอที่จะเดินหน้า?"

นี่คือคำถามที่สำคัญที่สุด คำตอบบอกคุณอย่างชัดเจนว่าต้องส่งมอบอะไร — ไม่ว่าจะเป็น demo, pilot, การอ้างอิง, หรือการรับประกัน

**ช่วงที่ 4: บริบทและข้อจำกัด (5-10 นาที)**

"มีอะไรอื่นอีกไหมเกี่ยวกับธุรกิจของคุณที่จะช่วยให้ผมเข้าใจว่าเหมาะหรือไม่?"

คำถามปลายเปิดนี้ดึงสิ่งที่คุณไม่ได้นึกจะถามออกมา:
- รูปแบบตามฤดูกาล ("Q4 เราวุ่นสุด นอกนั้นพอจัดการได้")
- ข้อกำหนดด้านกฎระเบียบ ("ต้องเก็บบันทึกไว้ 7 ปี")
- ปัจจัยทางวัฒนธรรม ("ทีมเราไม่ใช้อีเมล — ทุกอย่างอยู่บน LINE")
- การเปลี่ยนแปลงที่กำลังจะมา ("เราจะเปลี่ยน software บัญชี quarter หน้า")

## 3.4 Framework การสัมภาษณ์สำหรับ Internal Champion

ถ้าคุณอยู่ Track 4 — ผลักดัน AI adoption ภายในองค์กร — การสัมภาษณ์ของคุณจะมีหน้าตาต่างออกไป คุณไม่ได้คัดกรองผู้ซื้อ คุณกำลัง map ภูมิทัศน์ทางการเมือง

### Internal People Mapping

| ใคร | สิ่งที่เขาสนใจ | วิธีเข้าหา |
|-----|---------------------|-----------------|
| **หัวหน้าตรงของคุณ** | KPI ของเขา, ปริมาณงานทีม, ไม่อยากเสี่ยง | "ช่วยให้ทีมเราตี target ได้" |
| **VP / หัวหน้าแผนก** | Budget, ประสิทธิภาพกำลังคน, กลยุทธ์ | "สอดคล้องกับ [โครงการของบริษัท]" |
| **IT / Security** | Compliance, การจัดการข้อมูล, การ integrate | "ผมคิดเรื่อง security ไว้แล้ว — นี่คือแผน" |
| **ทีมที่ทำงาน manual** | ความมั่นคงในงาน, ปริมาณงาน, การยอมรับ | "ลดงานน่าเบื่อออก" |
| **ฝ่ายการเงิน** | เหตุผลด้านต้นทุน, ROI, ช่วงเวลาของ budget | "นี่คือตัวเลข — เราใช้อยู่ X เราจะประหยัด Y" |
| **ทีมข้างเคียง** | จะสร้างงานเพิ่มให้เขาไหม | "ไม่ต้องใช้อะไรจากทีมคุณเลย" |

### การสนทนา Pitch ภายใน

คุณไม่ได้ส่งอีเมลแบบ cold outreach แต่คุยกันตามทางเดินและระหว่างกินข้าว รูปแบบต่างกัน แต่เป้าหมายเดียวกัน: เข้าใจความเจ็บปวด map แรงจูงใจ สร้างความเข้าใจตรงกัน

**กับทีมที่ทำงาน manual:**

"เฮ้ ผมอยากรู้ว่าคุณจัดการ [งานเฉพาะ] อย่างไร ช่วยเล่าให้ฟังหน่อยได้ไหม?"

อย่าพูดถึง AI ก่อน อย่าพูดถึง automation แค่สนใจจริงๆ คนจะเปิดใจเมื่อรู้สึกว่าคุณสนใจงานของเขา แต่จะปิดกั้นเมื่อรู้สึกว่าคุณพยายามจะมาแทนที่

**ตรงนี้สำคัญมาก:** เมื่อคุณเสนอ automation ให้ frame ว่าเป็นการสนับสนุนทีมนี้ ไม่ใช่การแทนที่ "ตัวนี้จะจัดการส่วนกรอกข้อมูลซ้ำๆ เพื่อให้คุณโฟกัสกับงานที่ต้องใช้วิจารณญาณและความสัมพันธ์ได้" ทำให้เขาเป็นพันธมิตร ไม่ใช่ฝ่ายตรงข้าม รักษาศักดิ์ศรีของเขาไว้ นั่นไม่ใช่แค่กลยุทธ์ที่ดี — แต่เป็นสิ่งที่ถูกต้อง

**กับหัวหน้าของคุณ:**

"ผมดูกระบวนการของ [ทีม/งาน] อยู่ และคิดว่าน่าจะประหยัดได้ [X ชั่วโมง/สัปดาห์] ด้วย automation สักตัว อยากลอง pilot เล็กๆ — สองสัปดาห์ budget น้อย เกณฑ์วัดผลชัดเจน ถ้าเวิร์กก็ขยาย ถ้าไม่ก็ได้เรียนรู้"

องค์ประกอบหลัก:
- ความเสี่ยงต่ำ ("pilot เล็กๆ" "สองสัปดาห์" "budget น้อย")
- ผลตอบแทนชัดเจน ("ประหยัด X ชั่วโมง/สัปดาห์")
- มีทางออก ("ถ้าไม่เวิร์กก็ได้เรียนรู้")
- ไม่ได้ขอข้อผูกมัดใหญ่

**กับ IT / Security:**

บทสนทนานี้ต้องเกิดขึ้นตั้งแต่เนิ่นๆ ไม่มีอะไรฆ่า project ภายในได้เร็วเท่ากับตื่นเต้นสร้าง prototype แล้วโดน IT ปิดเพราะส่งข้อมูลไป external API โดยไม่ได้รับอนุมัติ

"ผมกำลังสำรวจ pilot project ที่จะใช้ [AI tool เฉพาะ] ก่อนไปต่อ อยากให้มั่นใจว่าคิดเรื่องการจัดการข้อมูลถูกต้อง นโยบายเราเรื่อง [cloud API / การประมวลผลข้อมูลภายนอก] เป็นอย่างไร?"

ขอความเห็น ไม่ใช่ขออนุญาต คนที่รู้สึกว่าถูกปรึกษาจะเป็นพันธมิตร คนที่รู้สึกว่าถูกข้ามจะกลายเป็นผู้ขัดขวาง

**กับ VP / ผู้ถือ budget:**

บทสนทนานี้มาทีหลัง — หลังจากคุณทำวิจัย คุยกับทีม และถ้าเป็นไปได้ มีผลลัพธ์ pilot แล้ว

"ผมพบว่า [ทีม] ใช้เวลา [X ชั่วโมง/สัปดาห์] กับ [งานเฉพาะ] ซึ่งมีต้นทุน [$Y/เดือน] ในค่าแรง ผมทำ pilot 2 สัปดาห์ automation ทำได้ [Z%] ของงาน Pilot มีค่าใช้จ่าย [$A] และประหยัดได้ [$B] ผมต้องการ budget [$C/เดือน] เพื่อทำต่อและขยายผล"

สังเกต: คุณไม่ได้ขออนุญาตสำรวจ คุณนำเสนอผลลัพธ์และขอทรัพยากรเพื่อทำสิ่งที่ได้ผลอยู่แล้วต่อไป ตอบตกลงได้ง่ายกว่ามาก

## 3.5 การเชื่อมโยงความเจ็บปวดกับคุณค่า

คุณสัมภาษณ์เสร็จแล้ว มีบันทึกเป็นหน้าๆ ตอนนี้ต้องแปลข้อมูลดิบเหล่านั้นเป็น value proposition ที่ชัดเจน

### สะพานจากความเจ็บปวดสู่คุณค่า

สำหรับทุกจุดเจ็บปวดที่พบ สร้างสะพานเชื่อม:

| ความเจ็บปวดของเขา | ผลกระทบที่วัดได้ | วิธีแก้ปัญหาของคุณ | คุณค่าที่สร้างได้ |
|-----------|-------------------|---------------|--------------|
| Front desk ใช้เวลา 3 ชม./วัน โทรหาคนไข้ | 15 ชม./สัปดาห์ x $25/ชม. = $1,500/เดือน | SMS/LINE เตือนอัตโนมัติพร้อมระบบติดตามอัจฉริยะ | ประหยัดค่าแรง $1,200/เดือน + ลดคนไม่มาตามนัด |
| อัตราไม่มาตามนัด 20% | 4 นัดพลาด/วัน x $150 เฉลี่ย = $600/วัน สูญเสีย | ระบบ waitlist เติมช่องว่างจากการยกเลิก | กู้รายได้ $6,000-8,000/เดือน |
| พนักงานหงุดหงิด ลาออกบ่อย | ค่าจ้างและฝึกคนใหม่ $3,000 x 2 ครั้ง/ปี | กำจัดงานที่ถูกเกลียดมากที่สุด | ลดค่าใช้จ่ายจากการลาออก $6,000/ปี |

**คุณค่ารวมต่อปี: ~$100,000+ สำหรับคลินิกทันตกรรมแห่งเดียว**

ตอนนี้ค่าบริการ $1,000/เดือน ของคุณก็ดูเหมือนเศษเงิน

## 3.6 การเขียน Value Proposition ในภาษาของลูกค้า

Value proposition ไม่ใช่ elevator pitch แต่เป็นข้อความที่ลูกค้าอ่านแล้วคิดว่า: "นี่คือสิ่งที่ต้องการเลย"

### สูตร

**สำหรับ [ลูกค้าเฉพาะ] ที่ [ความเจ็บปวดเฉพาะ] วิธีแก้ปัญหาของเรา [ผลลัพธ์เฉพาะ] ต่างจาก [ทางเลือกอื่น] ตรงที่ [จุดแตกต่างสำคัญ]**

### ตัวอย่างตาม Track

**สำหรับ developer ที่ขายให้คลินิกทันตกรรม:**
"สำหรับคลินิกทันตกรรมที่สูญเสียรายได้จากคนไข้ไม่มาตามนัด และเสียเวลาพนักงานกับการโทรศัพท์ เราทำ automation ระบบเตือนนัดหมายและติดตามคนไข้ผ่าน LINE และ SMS คลินิกที่ใช้ระบบของเราเห็นอัตราไม่มาตามนัดลดจาก 20% เหลือต่ำกว่า 8% ภายใน 30 วัน ต่างจาก software เตือนทั่วไปตรงที่เรา integrate กับระบบบริหารคลินิกที่มีอยู่แล้ว และจัดการลำดับการติดตามทั้งหมด — รวมถึงการนัดใหม่ — โดยพนักงานไม่ต้องแตะอะไรเลย"

**สำหรับ consultant ที่เพิ่ม AI ให้ลูกค้าที่มีอยู่:**
"สำหรับลูกค้าที่มีปัญหาเรื่องความเร็วในการตอบ lead เราเสนอระบบคัดกรอง lead ด้วย AI ที่ตอบคำถามใหม่ภายใน 3 นาที ตลอด 24 ชั่วโมง ลูกค้า pilot ของเราเห็นอัตราเปลี่ยน lead เป็นนัดหมายเพิ่มขึ้น 60% ระบบ integrate กับ CRM ปัจจุบัน — ไม่ต้องเรียนรู้ระบบใหม่"

**สำหรับ corporate champion ที่ pitch ภายใน:**
"ทีมบริการลูกค้าของเราใช้เวลา 30 ชั่วโมงต่อสัปดาห์ตอบคำถามเดิม 15 ข้อเกี่ยวกับการติดตามคำสั่งซื้อ ระบบตอบอัตโนมัติจัดการ 70% ของคำถามเหล่านี้ได้ ปลดปล่อยทีมให้โฟกัสกับปัญหาซับซ้อนที่ต้องใช้วิจารณญาณของคน จาก pilot 2 สัปดาห์ ประหยัดได้ $8,000/เดือน และปรับปรุงเวลาตอบจาก 4 ชั่วโมงเหลือ 3 นาที"

### กฎด้านภาษา

1. **ใช้คำศัพท์ของเขา** ถ้าเขาพูดว่า "คนไข้" อย่าพูดว่า "ผู้ใช้" ถ้าเขาพูดว่า "spreadsheet สีน้ำเงิน" ก็อ้างถึง "spreadsheet สีน้ำเงิน"

2. **นำด้วยผลลัพธ์ ไม่ใช่เทคโนโลยี** ผิด: "ระบบ NLP ที่ขับเคลื่อนด้วย AI ของเราประมวลผลการสื่อสารกับคนไข้" ถูก: "อัตราไม่มาตามนัดลดเหลือต่ำกว่า 8%"

3. **ใส่ตัวเลข** "ประหยัดเวลา" แพ้ "ประหยัด 15 ชั่วโมงต่อสัปดาห์" "เพิ่มรายได้" แพ้ "กู้คืน $6,000/เดือน จากนัดที่พลาด"

4. **ตอบความกลัว** เพิ่มตาข่ายนิรภัย: "Pilot 2 สัปดาห์ไม่มีข้อผูกมัด" "ทำงานคู่กับกระบวนการปัจจุบันจนกว่าคุณจะมั่นใจ" "ยกเลิกได้ทุกเมื่อ"

5. **ซื่อสัตย์เรื่องข้อจำกัด** "จัดการคำถามทั่วไปได้ 85% ปัญหาซับซ้อนยังส่งต่อให้ทีมของคุณ" ความซื่อสัตย์สร้างความไว้วางใจ การสัญญาเกินจริงทำลายมัน

## 3.7 การได้รับ Buy-In

การเข้าใจปัญหาและมี value proposition ไม่เพียงพอ คุณต้องมีคนตอบตกลง — ไม่ว่าจะเป็นลูกค้าเซ็นสัญญา VP อนุมัติ budget หรือทีมตกลงร่วม pilot

### สำหรับลูกค้าภายนอก (Track 1, 2, 3)

กระบวนการขายทั้งหมด — จาก proposal ถึง close — อยู่ในบทที่ 5 สิ่งที่สำคัญตรงนี้คือการเข้าใจว่าแต่ละคนในกระบวนการตัดสินใจต้องได้ยินอะไรถึงจะตอบตกลง

**สร้างความเร่งด่วนโดยไม่กดดัน** "คุณใช้จ่ายประมาณ $1,500/เดือน กับกระบวนการ manual นี้ ทุกเดือนที่รอคืออีก $1,500 ผมทำ pilot ให้ทำงานได้ภายในสองสัปดาห์"

**ลดความเสี่ยง** กลยุทธ์ free pilot (รายละเอียดในบทที่ 5.2) เป็นเครื่องมือที่ทรงพลังที่สุดตรงนี้

**ทำให้ขั้นตอนต่อไปเล็กจิ๋ว** อย่าขอให้ "ผูกพันสัญญา 12 เดือน" ขอให้ "นัดประชุม kickoff 30 นาที" ทุกก้าวเล็กๆ สร้างโมเมนตัม

**จัดการกับผู้ขัดขวางตรงๆ** ถ้าผู้จัดการสำนักงานชอบแต่เจ้าของยังไม่เห็น: "ถ้าผมทำสรุป ROI 1 หน้าสำหรับ [เจ้าของ] จะช่วยไหม? ผมทำให้ได้ภายในพรุ่งนี้"

### สำหรับ project ภายใน (Track 4)

การได้ buy-in ภายในองค์กรยากกว่าการขายภายนอก เพราะโครงสร้างแรงจูงใจซับซ้อนกว่า

**ขั้นตอนที่ 1: สร้างฐานหลักฐาน** ก่อน pitch ใครที่มีอำนาจ รวบรวมข้อมูลให้ครบ:
- ชั่วโมงที่ใช้กับกระบวนการ manual
- ต้นทุนของชั่วโมงเหล่านั้น
- อัตราข้อผิดพลาดของกระบวนการปัจจุบัน
- ผลกระทบของข้อผิดพลาด (ลูกค้าร้องเรียน เวลาแก้ไข ความเสี่ยงด้าน compliance)

**ขั้นตอนที่ 2: ทำ guerrilla pilot** ถ้าสร้างหรือตั้งค่า prototype ได้โดยไม่ต้องขออนุมัติ budget (ใช้ free tier, เวลาส่วนตัว, ข้อมูลที่ไม่ใช่ production) ก็ทำเลย ผลลัพธ์จริงน่าเชื่อกว่าการคาดการณ์บนกระดาษ 10 เท่า

**ขั้นตอนที่ 3: หา champion ของคุณ** คุณต้องมีคนที่มีอำนาจซึ่งจะสนับสนุน project ของคุณในที่ประชุมที่คุณไม่ได้อยู่ เขาไม่ต้องเข้าใจเทคโนโลยี แค่เข้าใจผลลัพธ์และยินดีพูดว่า "ผมสนับสนุนเรื่องนี้" ในที่ประชุม budget

Champion มักจะเป็นหัวหน้าของคุณ (ถ้าเข้าใจคุณค่าและมีอำนาจจัดสรร budget) หรือ Director / VP ที่พูดเรื่อง "นวัตกรรม" หรือ "digital transformation" อยู่เป็นประจำ

**ขั้นตอนที่ 4: นำเสนอผลลัพธ์ ไม่ใช่แผน**

"ผมทำ pilot 2 สัปดาห์ นี่คือผลลัพธ์ ต้องการ $X/เดือน เพื่อทำต่อและขยาย"

ได้ผลดีกว่า "ผมมีไอเดีย ขอ budget สำรวจได้ไหม?" มากนัก

**ขั้นตอนที่ 5: ทำให้ตอบตกลงง่าย**
- ขอเงินเริ่มต้นน้อยๆ ($500-1,000/เดือน ไม่ใช่ $50,000/ปี)
- เสนอช่วงทดลองที่ชัดเจน (3 เดือน มี review)
- มี kill switch ("ถ้าไม่ตี [ตัวชี้วัดเฉพาะ] ภายในเดือนที่ 3 ก็หยุด")
- เชื่อมกับโครงการที่มีอยู่ ("สนับสนุนเป้าหมายจากการประชุมพนักงานครั้งล่าสุด")

## 3.8 การกำหนด Scope: รวมอะไร ยังไม่ทำอะไร

คุณได้ buy-in แล้ว ก่อนเริ่มสร้าง ต้องกำหนด scope — และการกำหนด scope สำคัญที่สิ่งที่ไม่รวม พอๆ กับสิ่งที่รวม

### เอกสาร Scope

ทุก project ต้องมีเอกสาร scope หนึ่งหน้า:

1. **Workflow ไหนที่จะทำ automation?** (หนึ่ง workflow ไม่ใช่สาม)
2. **อะไร trigger automation?** (อีเมลใหม่, การกรอกแบบฟอร์ม, ตามตารางเวลา)
3. **Automation ทำอะไร?** (ขั้นตอนที่ 1-5 ที่ชัดเจน)
4. **ผลลัพธ์คืออะไร?** (อีเมล, แถวใน spreadsheet, แจ้งเตือน, รายงาน)
5. **อะไรที่ไม่รวม?** (รายการชัดเจน)
6. **วัดผลสำเร็จอย่างไร?** (ชั่วโมงที่ประหยัด, อัตราข้อผิดพลาด, เวลาตอบ)
7. **ลูกค้าต้องให้อะไร?** (สิทธิ์เข้าถึง, ข้อมูล, คนประสาน)
8. **Timeline?** (โดยทั่วไป 2-3 สัปดาห์สำหรับการ deploy เริ่มต้น)

### รายการ "Phase 2"

ระหว่างทุกการสนทนา ลูกค้าจะพูดถึงสิ่งที่อยากมี อย่าตอบว่าไม่ ตอบว่า "นั่นเป็นไอเดียดีสำหรับ Phase 2"

รายการนี้มีประโยชน์สามอย่าง:
1. ลูกค้ารู้สึกว่าถูกรับฟัง (ไม่ถูกปฏิเสธ)
2. คุณมี roadmap สำหรับ upsell ในภายหลัง
3. คุณปกป้อง scope ปัจจุบันจากการขยายตัว

### รวมอะไรใน Phase 1

ใช้กฎ 80/20 อย่างเข้มงวด:

- **รวม:** Workflow เดียวที่เป็นสาเหตุ 80% ของความเจ็บปวด
- **รวม:** Error handling พื้นฐาน (เมื่อมีปัญหา ให้แจ้งคน — ไม่ต้องเดา)
- **รวม:** รายงานรายสัปดาห์ที่แสดงว่า automation ทำอะไรบ้าง (เพื่อ justify การลงทุน)
- **รวม:** คิวตรวจสอบโดยคนสำหรับ output ที่ confidence ต่ำ

- **ยังไม่ทำ:** Dashboard หรือ UI (รายงานทางอีเมลรายสัปดาห์เพียงพอ)
- **ยังไม่ทำ:** รองรับหลายภาษา (เริ่มด้วยภาษาหลักก่อน)
- **ยังไม่ทำ:** Integrate กับทุกเครื่องมือที่ใช้ (เริ่มกับตัวที่สำคัญที่สุด)
- **ยังไม่ทำ:** Mobile app (อย่า... ไม่ใช่ใน Phase 1)

สร้างของที่ใช้ได้ตอนนี้ สร้างเวอร์ชันสมบูรณ์แบบทีหลัง

### กับดักการกำหนด Scope

**กับดัก "อีกอย่างเดียว":** แต่ละคำขอดูเหมือน "แค่ 15 นาที" สิบอันรวมกันเป็นสัปดาห์ ยืนหยัด: "ไอเดียดี — ใส่รายการ Phase 2"

**กับดัก "ทำให้สมบูรณ์แบบ":** คุณอยากสร้างของที่ภูมิใจ หักห้ามใจ ความภูมิใจมาจากการที่คนนำไปใช้จริง ไม่ใช่จากสถาปัตยกรรม

**กับดัก "เขาไม่รู้ว่าต้องการอะไร":** ไม่เป็นไร — นั่นคือเหตุผลที่คุณเป็นผู้เชี่ยวชาญ เสนอ scope เฉพาะจากการสัมภาษณ์ แล้วให้เขาตอบสนอง การแก้ไข proposal ง่ายกว่าการสร้างจากศูนย์

## สิ่งที่ต้องส่งมอบ — บทที่ 3

เมื่อจบบทนี้ คุณควรมี:

1. **บันทึกสัมภาษณ์เชิงลึก** จากลูกค้าเป้าหมาย 2-3 ราย หรือคนภายในองค์กร — ความเจ็บปวดที่ map ไว้ ความกลัวที่ระบุ กระบวนการตัดสินใจที่เข้าใจแล้ว
2. **สะพานจากความเจ็บปวดสู่คุณค่า** — แสดงผลกระทบที่วัดได้ของแต่ละจุดเจ็บปวด และคุณค่าที่วิธีแก้ปัญหาของคุณสร้างได้
3. **Value proposition ที่เขียนแล้ว** — ในภาษาของลูกค้า พร้อมตัวเลขเฉพาะ
4. **กลยุทธ์การได้ buy-in** — รู้ว่าใครต้องตอบตกลง อะไรจะทำให้เขาตอบตกลง และข้อเสนอลดความเสี่ยงของคุณ
5. **เอกสาร scope Phase 1** — หนึ่งหน้า หนึ่ง workflow ตัวชี้วัดความสำเร็จชัดเจน สิ่งที่ไม่รวมชัดเจน

**สิ่งที่คุณพกไปบทที่ 4:** ภาพชัดเจนว่าจะสร้างอะไร สร้างให้ใคร ความสำเร็จหน้าตาเป็นอย่างไร และมีข้อตกลงจากคนที่สำคัญ

คุณยังไม่ได้เขียน code สักบรรทัด นั่นตั้งใจ คนสร้างที่รีบเขียน code สร้างของผิด ผู้ดำเนินการที่ใช้เวลาเข้าใจก่อน สร้างของถูก — และสร้างเร็วกว่า เพราะไม่ต้องเดา

ตอนนี้คุณพร้อมสร้างแล้ว บทที่ 4 ครอบคลุมสถาปัตยกรรม, tech stack, และ build sprint 72 ชั่วโมง
